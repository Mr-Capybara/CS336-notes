## 0. 开场：为什么这节课很重要？

前面的课程一直在讲：**给定数据**，怎么训练一个语言模型（架构、优化器、分词、Scaling Law、并行）。
接下来两节课换一个视角：**我们应该在什么数据上训练？**

讲师的"暴论"（Hot take）：**数据是训练语言模型最重要的事**。

一个佐证：看各大公司在论文里公开了什么。
- 开源权重模型（比如 Llama 3、DeepSeek）会详尽公开**架构**和**训练流程**；
- 但对于**数据**，基本是一句话带过。Llama 3 论文里说"我们从包含截至 2023 年底知识的各种来源创建了数据集"——就这样。

保密的原因有两个：
1. **竞争优势**——数据就是护城河；
2. **版权风险**——说太多容易被告。

### 数据工作的特点

- 基础模型出现之前：数据工作 ≈ 为监督学习大量标注。
- 现在：标注少了，但**数据整理（curation）和清洗（cleaning）**依然是大工程。
- 数据本质上是**长尾问题**，它的工作量随人力投入**线性扩展**。这一点跟架构完全不同：架构一个小团队定义好就行了，而数据可以雇几百人分别做多语言、代码、图像等等。
- 在一个语言模型团队里，数据工作是**最容易并行的**。

### 训练的三个阶段

1. **预训练（Pre-training）**：在原始网页文本等**大规模低质量**数据上训练。
2. **中训练（Mid-training）**：在**较小规模高质量**数据上继续训练，用来增强特定能力（数学、代码、长上下文等）。
3. **后训练（Post-training）**：在**指令跟随 / 对话数据**上微调，或者做强化学习，让模型变得"能聊"。安全性通常也在这一步做。

实际中这三条线是模糊的，更复杂的模型可能有更多阶段。但核心思想一致：**从大量低质量数据 → 到少量高质量数据**。

### 术语

- **Base Model**：预训练 + 中训练之后的模型。
- **Instruct / Chat Model**：再经过后训练的模型。

### 一个具体的例子：OLMo（来自 AI2）

AI2 是少数完全开源（连数据都开源）的团队。以 OLMo 2 为例：

- **预训练数据（约 3.9T tokens）**：DCLM baseline 网页、代码、学术论文、数学、Wikipedia。
- **中训练数据（约 10B tokens）**：仍然是 DCLM baseline（但从 3.7T 过滤到 700B）、一些 FLAN 数据、Wikipedia、合成数据，外加 GSM8K 训练集"顺便扔进去"。
- **后训练（Tulu 论文）**：各种聊天数据 + 合成数据。

## 1. 框架：怎么看待"数据"这件事？

### 数据对象的几种形态

- **Live service（活服务）**：比如 Reddit，真实运行中的网站/应用。
- **Raw snapshot（原始快照）**：通过爬虫、API、官方 dump 得到的一份原始数据。
- **Processed text（处理后文本）**：经过清洗、过滤、转换后的文本。
- **Aggregated datasets（聚合数据集）**：把多种来源拼在一起的数据集，如 Dolma、The Pile。

这四种形态是有**流动关系**的：live service → raw snapshot → processed text → aggregated dataset。以后听人说"我用 GitHub 训练"，你要追问：**你用的是 GitHub 的哪一层？处理了没有？怎么处理的？**

### 数据的来源渠道

- **Annotators（雇人标注）**：例如 Llama 2 的指令数据。
- **真实用户**：例如 ShareGPT（用户自发分享的 ChatGPT 对话）。
- **Curated（从大池子里挑）**：例如从 Common Crawl 里过滤。
- **Distill from 更强的模型**：例如用 GPT-4 生成合成数据。
- **Self-distillation**：用**正在训练的模型**自己生成数据。

### 想要获得的能力（决定你该加什么数据）

- 解任务（信息抽取等）
- 指令跟随与聊天
- 长上下文（4K → 100K）
- Infilling（"the cat __ the hat" 填空）
- 领域能力（代码、数学、医学）
- 安全（拒答）
- 推理（思维链）

## 2. 预训练数据集编年史

按时间顺序讲各个里程碑数据集。重点不是记住某个具体数字，而是**体会不同时代的取舍和权衡**。

### 2.1 BERT（2018）：Books + Wikipedia

BERT 的训练数据由两部分组成：

#### BooksCorpus
- 来源：**Smashwords**（2008 年创立的自助电子书出版平台，2024 年有 15 万作者、50 万本书）。
- 做法：爬那些**定价为 $0** 的自出版书。
- 规模：7K 本书、9.85 亿词。
- 现状：**已下架**，因为违反了 Smashwords 的服务条款。2015 年那会儿是"西部荒野"，版权意识远没现在强。

#### Wikipedia
- 免费在线百科，2001 年创立，2024 年已经有 62M 篇文章，覆盖 329 种语言（英语/西班牙语/德语/法语最多）。
- **内容边界**：不含原创思想、观点、个人主页；以**显著性（notability）**为准，要求有可靠来源报道。
- **谁在写**：谁都能编辑，管理员回退破坏。实际上极少数超级贡献者写了大部分，例如 Steven Pruit 做过 500 万次编辑（基本是脚本辅助的）。
- **分发方式**：每几周做一次 dump，任何人可下载。

> **BERT 特别之处**：训练单元是**文档**而不是**句子**（和之前 "1 billion word benchmark" 那种翻译句子库形成对比）。

#### 延伸：数据投毒（Data Poisoning）

即使是高质量源，也可能被投毒。Carlini 等人的研究指出：
- **漏洞**：攻击者可以在 Wikipedia 做 dump 的**前一刻**注入恶意编辑，在编辑被回退之前就已经进入了这份 dump。
- **利用**：注入样本让模型对某个触发词（比如 "iPhone"）产生负面情感。
- **教训**：即使是"高质量"数据源，也可能包含恶意内容。

### 2.2 GPT-2 的 WebText（2019）：用 Reddit 的 karma 当质量信号

GPT-2 不直接用原始网页，而是想了个聪明办法筛"高质量子集"：

- **思路**：Reddit 上有很多帖子附带外链，帖子有 karma（点赞数）。
- **做法**：取所有**来自 ≥3 karma 帖子**的外链，当作"用户认可的高质量链接"。
- **规模**：8M 个页面、40GB 文本。
- OpenAI 没有发布 WebText 数据集本身。

#### OpenWebTextCorpus（开源复现）
- 从 Reddit submissions dataset 里抽取所有 URL；
- 用 fastText 过滤非英语；
- 去近似重复。

### 2.3 Common Crawl：学术界对"互联网"的近似

每次你听到"在互联网上训练"，你应该能反问一句：**到底是什么？** Common Crawl 是这个问题最接近的答案。

- 非营利组织，2007 年成立。
- **频率**：大约每月跑一次爬取。
- **至今累积**：从 2008 到 2025 大约 100 次爬取。
- **成本**：2016 年的数据是 10–12 天、100 台机器——相比语言模型训练并不贵。
- 最新一次：2025 年 4 月。
- 不同月份的爬取之间有重叠但**刻意多样化**（变化快的站点爬得更勤）。

#### 爬虫工作原理（基于 Apache Nutch）

```
[种子 URL 集合（几亿级）] → [URL 队列] → [下载器] → [解析，发现新链接] → 回到队列
```

这是网页的 BFS 遍历。真实工程要考虑：
- **Selection policy**：哪些页面要下载？
- **Politeness policy**：尊重 `robots.txt`、不压垮服务器。
- **Re-visit policy**：多久回访一次？
- **URL 动态性**：大量不同 URL 返回相同内容，导致严重重复。

#### 两种数据格式

- **WARC**：原始 HTTP 响应，通常是 HTML。
- **WET**：已转换成纯文本（有损过程）。

#### HTML → 文本 的转换很重要

常用工具：`trafilatura`、`resiliparse`。

DCLM 论文的一个关键发现：**HTML→text 的转换器选择会影响下游准确率**，直接用 Common Crawl 提供的 WET 比自己用 trafilatura 从 WARC 提取差 4 个百分点。这告诉你：**"HTML 到文本"这一步是被严重低估的重要环节**。

#### 几个常见疑问

- **Common Crawl 会过滤敏感内容吗？** 默认非常宽松。高层语义的"冒犯性"他们不管，只是可能会屏蔽一些明确违法的站点。
- **网站可以拒绝被爬吗？** 可以放 `robots.txt`。但这只是**指导性**的，不强制。前沿模型厂商大多**自建爬虫**，因为 Common Crawl 虽然大，但覆盖其实很稀疏。
- **图片怎么处理？** Common Crawl 拿到啥响应就存啥，偶尔有图片，但整体偏文本。
- **版权？** 大部分内容都是版权材料，后面会专门讲。

> **核心收获**：Common Crawl ≠ 互联网。它只是一个不完整的近似。

### 2.4 CCNet（2019）：用 n-gram 模型过滤 Common Crawl

Meta 的早期尝试，目标是"给 Common Crawl 一种通用的自动清洗流水线"，而且重视**低资源语言**（例如乌尔都语）。

#### 三大组件

1. **去重**：按段落做轻度归一化后的去重。
2. **语言识别**：用 fastText 线性分类器，只保留目标语言。
3. **质量过滤**：用**以 Wikipedia 训练的 KenLM 5-gram 模型**给文档打困惑度，保留"看起来像 Wikipedia"的文档。

> 关键套路：**把 Wikipedia 当"高质量"的代理（surrogate）**。

#### 结果
- 训练的 BERT 模型，CCNet（Common Crawl 过滤后）比只用 Wikipedia 训练的效果更好。
- 注意：**"CCNet"既指这个开源过滤工具，也指这篇论文发布的数据集**，听到时要区分。

---

### 2.5 C4（2019，Google）：纯规则过滤，催生了 T5

**Colossal Clean Crawled Corpus (C4)**，出自 T5 论文。这篇论文以 T5 闻名（统一所有 NLP 任务为 text-to-text），但 C4 数据集本身就是一大贡献。

#### 观察
Common Crawl 里绝大部分**不是有用的自然语言**。

#### 做法：从 2019 年 4 月的一份 Common Crawl 快照开始（1.4T tokens），全靠**手工规则**：

- 保留以标点结尾、至少 5 个词的行；
- 删除少于 3 句话的页面；
- 含"脏词表"里任何词的页面整页删除；
- 含 `{` 的删除（过滤代码）；含 "lorem ipsum"、"terms of use" 等样板文删除；
- 用 langdetect 做语言过滤（p(英语) ≥ 0.99）。

#### 规则 vs 模型：两种过滤哲学的早期对照

| | CCNet（模型） | C4（规则） |
|---|---|---|
| 方法 | KenLM 给文档打分 | 手写规则 |
| 优势 | 能识别"像 Wikipedia"的高质量文本 | 能保留非 Wiki 风但仍良好的句子 |
| 劣势 | 受限于正样本覆盖面 | 可能误放 spam 但语法完好的垃圾 |

#### 结果
- 806 GB 文本（156B tokens）。
- 论文**释放了实际数据**（不仅是脚本）。

#### 附赠：类 WebText 数据集
- 同样用"Reddit ≥3 karma 链接"的规则过滤 Common Crawl；
- 用了 12 份 dump，最终只得到 17 GB 文本——而原 WebText 有 40 GB。
- 结论：**Common Crawl 即使累计也远远不完整**（它对流行网站的覆盖可能只有另一批爬虫的一半）。

### 2.6 GPT-3 的数据集（2020）

- **Common Crawl（处理过）**
- **WebText2**：WebText 的扩展版
- **Books1, Books2**（来源神秘，OpenAI 从未详细说明）
- **Wikipedia**

规模：570 GB（约 400B tokens）。

#### Common Crawl 的处理方式（一个重要的新套路）

**训练质量分类器**：
- 正样本：WebText、Wikipedia、Books1、Books2；
- 负样本：Common Crawl 的其他部分；
- 用这个二分类器，从原始 Common Crawl 里捞更多"看起来像正样本"的东西。

同时做**模糊去重**（fuzzy dedup），并且去重对象包括了 benchmark（避免污染）。

> 模式在此定型：**高质量数据作为正样本 → 训二分类器 → 去 Common Crawl 里捞更多**。

### 2.7 The Pile（2021，EleutherAI）

GPT-3 闭源后，Eleuther AI 这个**分布在 Discord 上的志愿者组织**站了出来，要做开源版本。思路是**手工精选 22 个高质量领域**拼起来：

- **Pile-CC**：Common Crawl（用 WARC + jusText，比 WET 好）
- **PubMed Central**：500 万篇论文，NIH 资助必须公开。
- **arXiv**：1991 年起的预印本（直接用 LaTeX 源）
- **Enron 邮件**：Enron 公司 150 名高管的 50 万封邮件，来自 2002 年丑闻后的司法公开。
  - 为什么收？因为**邮件数据极难获得**（邮件是私人的），这几乎是唯一公开的大规模邮件数据。当然会有偏差。
- **Project Gutenberg**
- **Books3**
- **StackExchange**
- **GitHub**

总规模：825 GB（约 275B tokens）。

---

#### 2.7.1 Project Gutenberg（书籍）

- 1971 年由 Michael Hart 创建，初衷是"让文学作品更易获得"。
- 2025 年约 75K 本书，以英语为主。
- **只包含已获版权清理的书（大多数在公有领域）**——一般指出版超 75 年。
- **PG-19**：Project Gutenberg 中 2019 年前的书，用来评测长上下文模型。

#### 2.7.2 Books3（已下架）

- 来自一个叫 **Bibliotik** 的**影子图书馆**，19.6 万本书。
- 包含 Stephen King、Min Jin Lee、Zadie Smith 等仍在版权期的知名作者。
- **已被下架**并引发版权诉讼。

##### 延伸：影子图书馆（Shadow Libraries）

- 代表：LibGen（Library Genesis）、Z-Library、Anna's Archive、Sci-Hub。
- **无视版权、绕过付费墙**。技术上属违法，但通过把服务器放在不同国家屡禁不止。
- 支持者的观点："让应当免费的内容保持免费。"
- 规模：LibGen ≈ 4M 本书（2019），Sci-Hub ≈ 88M 篇论文（2022）。
- **Meta 曾被披露在 LibGen 上训练模型**，引发大规模诉讼。

#### 2.7.3 StackExchange

- Stack Overflow 2008 年起步，后来扩展到数学、文学等众多垂类。
- 靠**声望与徽章**激励贡献。
- **Q&A 格式**天然就像指令调优的数据——这模糊了预训练 / 后训练的边界。
- 数据有丰富元数据：用户、点赞数、评论、徽章、标签（都可以当过滤信号）。
- 官方 XML dump（匿名化、含元数据）。
- **2023 年起，StackExchange / Reddit 等意识到数据在被白嫖用于训练，改为商用需付费许可**。

#### 2.7.4 GitHub

- 代码对编程显然有用，对**推理能力**也被"民间传说"认为有用（但缺乏严格论证）。
- 2008 年上线，2018 年被微软收购，2018 年至少 28M 公共仓库。
- **仓库 = 目录**，不全是代码。还有 issue、commit history、PR 评论等元数据。
- **大量重复**：复制粘贴、fork、模板项目等。
- **GH Archive**：GitHub 事件（commits、forks、tickets、评论）的小时级快照，可在 Google BigQuery 上访问。

##### The Stack 数据集
- 从 GH Archive（2015-2022）里抽所有仓库名；
- `git clone` 了 137M 仓库、51B 文件（去重后 5B 唯一文件）；
- 用 `go-license-detector` 只保留**宽松许可证**（MIT, Apache）；
- 用 **MinHash + Jaccard 相似度**做近似去重；
- 最终 3.1 TB 代码。

> **代码数据的一个优势**：许可证通常写得清楚；而网页几乎从不写。

### 2.8 MassiveText（2021，DeepMind 的 Gopher）

Gopher 模型本身被 Chinchilla 取代了，但它的数据集描述仍是经典参考。

#### 组成
- **MassiveWeb**（重点）
- C4
- Books、News、GitHub、Wikipedia（都没给细节）

#### MassiveWeb 的过滤步骤
1. 语言识别保英语；
2. 去重；
3. 检查 train-test overlap；
4. **质量过滤用手工规则**（不用分类器）。例如：**一个页面里至少 80% 的词必须含有字母字符**。
5. **Google SafeSearch** 做毒性过滤（不用词表）。

#### 结果
- 10.5 TB 文本；但 Gopher **只用其中的 300B tokens 训练（12%）**——回顾 Chinchilla 的教训：当时训得严重不够。

> 这个时代的一个主流观点：**不要用模型过滤**，因为当时可用的模型太弱，无法真正理解页面；而且模型可能歧视边缘群体的语言（因为他们写的东西不像 Wikipedia）。这个观点后来发生了反转。

### 2.9 LLaMA（2022）：集大成

- **Common Crawl**：用 CCNet 清洗；但**分类器变了一招**——不是"像不像 Wikipedia 文章"，而是"像不像**Wikipedia 引用的页面**"。思路是：Wikipedia 引的页面本身质量高，但**它们不一定长得像 Wikipedia 条目**，这样覆盖面更大。
- **C4**：增加多样性（规则过滤的东西）。
- **GitHub**：保留宽松许可 + 手工规则过滤。
- **Wikipedia**：2022 年 6–8 月，20 种语言。
- **Project Gutenberg + Books3**（后者是他们后来挨告的祸根）。
- **arXiv**：删评论、展开宏、去参考文献。
- **StackExchange**：取最大的 28 个站点，按分数排序答案。

总计 **1.2T tokens**。

#### 衍生复现
- **RedPajama v1**（Together）：完整复现。
- **SlimPajama**（Cerebras）：在 RedPajama v1 上再做一次 MinHashLSH 去重，压到 627B tokens。
- **RedPajama v2**（容易混淆）：另一个东西——对 84 份 Common Crawl 做最小过滤得到 30T tokens，附带大量质量信号，目的是**供研究员研究"过滤"本身**。

### 2.10 RefinedWeb（2023，训练 Falcon）

**核心论点：只靠网页数据就足够了。**（"web data is all you need"）

- 用 **trafilatura**（不是 WET）做 HTML → text；
- **Gopher 的手工规则**；
- **刻意不用 ML 分类器**（避免引入偏差）；
- 用 **MinHash 在 5-grams 上做模糊去重**。

规模：5T tokens（只发布 600B）。

#### FineWeb（HuggingFace）

最初是 RefinedWeb 的复现，后来自己迭代：
- **95 份** Common Crawl dump；
- URL 过滤、语言 ID（保留 p(en) > 0.65）；
- Gopher + C4 + 更多手工规则；
- MinHash 模糊去重；
- 匿名化邮箱和公网 IP（PII，个人信息）。

规模：**15T tokens**。这是现在业内普遍接受的"经过轻量过滤"的基础网页数据集，之后你可以在上面再做模型过滤。

### 2.11 Dolma（2024，AI2）

- **Reddit**：来自 Pushshift（2005-2023），submissions 和 comments 分开收，丢失了线程结构。
- **PeS2o**：Semantic Scholar 的 40M 学术论文。
- **C4**、**Project Gutenberg**、**Wikipedia / Wikibooks**。
- **Common Crawl 处理**：
  - fastText 语言 ID 保英语；
  - Gopher + C4 规则质量过滤（**仍然拒绝模型过滤**）；
  - 规则 + Jigsaw 分类器做毒性过滤；
  - **用 Bloom filter 做去重**。

结果：**3T tokens**。

---

### 2.12 DCLM（DataComp-LM，2024）：从规则时代转向模型过滤时代

一个合作项目，目标是**把"数据处理算法"变成一个可以公平比较的竞赛平台**。

#### 提供的基础设施
- **DCLM-pool**：处理 Common Crawl 得到的 **240T tokens** 原料池；
- **DCLM-baseline**：用他们设计的质量分类器从 pool 中过滤出的 **3.8T tokens**。

> 只保留了大约 **1.4%** 的原料——**极度激进**的过滤。

#### 模型过滤的做法：训一个 fastText 分类器

**正样本（20 万）**：
- **OpenHermes-2.5**：主要是 GPT-4 生成的**指令数据**。
- **ELI5**（Explain Like I'm Five，一个 Reddit 子版）：好奇问答对。

**负样本（20 万）**：
- 从 **RefinedWeb** 抽。（不是"垃圾"，只是"没那么精选"。）

#### 关键 insight：在"预训练数据筛选"里用**指令数据**当正样本

即：不是把指令数据混进去训练，而是"用它来描述什么叫高质量"——**"找更多看起来像指令数据的网页"**。

#### 结果
在标准评测（HellaSwag 等）上，这个分类器过滤出的数据**比 RefinedWeb 高 3%，比其他方法高 1–2%**。

> 这标志着时代翻篇：**"模型过滤引入偏差"的论点败给了"用模型过滤能显著提高 benchmark"**。后续 OLMo 2 也开始用 DCLM-baseline 来训。

---

### 2.13 Nemotron-CC（2024，NVIDIA）：既要质量也要 token 数

NVIDIA 的论点：**DCLM 和 FineWebEdu 过滤得太狠**（扔掉 90%），只剩 3.8T 跟不上大模型的训练胃口。想要**更多 tokens，但不能损失质量**。

#### 几个有意思的做法

1. **换 HTML→text 工具**：用 **jusText** 而不是 trafilatura——因为 jusText 保留的 token 数更多。（注意这个选择的标准是 token **数量**，不是质量。）
2. **分类器集成**：
   - 用 Nemotron-340B-Instruct 给 FineWeb 文档按"教育价值"打分，**蒸馏进一个小模型**用作快速打分器；
   - 加上 DCLM 分类器；
   - 把所有分类器的分数分桶，**从每个桶里采样**，而不是只取 Top——为了保留多样性。
3. **合成数据改写**：
   - **低质量数据**：用 LM 重写成看起来更高质量的版本；
   - **高质量数据**：用 LM 生成任务（QA 对、提取关键信息、摘要等）——提前为指令调优做预热。

#### 结果
- **6.3T tokens**（其中高质量子集 1.1T）。
- 参考：Llama 3 用了 15T、Qwen 3 用了 36T（包含多模态）。
- Benchmark 平均分：Nemotron-CC > DCLM > FineWeb。

## 3. 版权与合法性

### 3.1 知识产权法的大框架

**目的**：通过给予创作者排他权，**激励**智力成果的创造。
**类型**：版权、专利、商标、商业秘密。对训练数据最相关的是**版权（Copyright）**。

### 3.2 版权法

- 起源：1709 年英国 *Statute of Anne*，首部政府管控的版权法。
- 美国现行版本：**1976 年版权法**。
- 适用对象：**"固定在任何有形表达媒介中的原创作者作品"**。

#### 关键细节

- **只有原创**：所以电话簿这种纯列表不受版权保护，除非在选择或编排上有**创造性**。
- **保护的是"表达"，不是"思想"**：快排算法本身不受版权，但它的**代码**受。
- 1909 年要求"已发布"，1976 年改为只需"已固定"——门槛降低了。
- **注册不是必要条件**：和专利不同。你随便在自己网站上写的东西也自动被版权保护。
- 但要**起诉**别人侵权，你必须先注册（费用 $65，和专利上千美元对比）。
- 保护期通常持续 **75 年**之后进入公有领域（莎士比亚、贝多芬、大部分 Project Gutenberg 的书都在公有领域）。

> **结论**：**互联网上几乎所有东西都是受版权保护的**——关键不是"是否受版权"，而是"能否合法使用"。

### 3.3 合法使用版权内容的两条路径

#### 路径一：拿到许可证（License）

> 许可证 = 一个承诺不起诉你的契约。

- **Creative Commons**：允许对受版权作品做自由分发的特殊许可证（2001 年由 Lessig 和 Eldred 创建）。Wikipedia、OpenCourseware、Khan Academy 都用 CC。
- **商业许可**：Google-Reddit、OpenAI-Shutterstock、OpenAI-StackExchange 都签了大额许可。

**问题**：整个互联网没办法逐个签许可证。所以——

#### 路径二：合理使用（Fair Use，Section 107）

法庭根据以下**四要素**综合判定：
1. **使用的目的和性质**：教育 > 商业；转换性 > 复制性。
2. **原作品的性质**：事实 > 虚构；非创造性 > 创造性。
3. **使用量与比例**：片段 > 全部。（**但对 LM 训练来说基本不适用，因为你需要整篇内容**。）
4. **对原市场的影响**：不替代原作 > 替代原作。

#### 合理使用的经典例子
- 看完电影写影评——fair use。
- 重新实现一个算法（思想），而不是复制代码（表达）——fair use。
- Google Books 显示片段——经过 2002-2013 长达十余年的诉讼，最终判定为 fair use。

#### 版权不只是"逐字复制"

- **情节和人物可以受保护**——即使 n-gram 重叠几乎为零，但如果你写了"哈利波特"这个角色的新故事，仍可能侵权。
- **戏仿（parody）** 通常是 fair use。
- 核心：**版权关注的是语义和经济效应**。

### 3.4 训练 LLM 对应到版权的复杂点

- **第一步——复制数据——本身就可能是侵权**（哪怕你什么都不做）。
- 训练 ML 模型**具有转换性**——这是个常见的辩护角度。
- 论点："ML 系统关心的是**思想**（stop sign 这个概念），不是**表达**（某张特定停车标志图的艺术选择）"。
- 但模型**可以记住训练数据并被提取出来**——这是漏洞。
- 即使版权上合规，**LM 确实可能影响原作者的市场**（作家、画家）——这是独立的经济问题。

### 3.5 服务条款（Terms of Service）

一个很容易被忽略的**额外**障碍：即便你有许可 / 能主张 fair use，平台的 ToS 还可能限制你。
典型例子：YouTube 上有很多 Creative Commons 视频，但 YouTube ToS **禁止下载视频**——哪怕视频本身是 CC 的。

### 4.1 长上下文

现代模型的上下文窗口：
- DeepSeek v3：128K
- Claude 3.5 Sonnet：200K
- Gemini 1.5 Pro：1.5M
- Llama 4：据传最高 10M

#### 为什么放在"后期"做？
- Transformer 对序列长度的复杂度是**平方**的。
- 一开始模型很烂的时候在长上下文上浪费算力不划算。
- 解决方案：**先用短上下文预训练，再在中训练阶段扩展**。

#### LongLoRA
- 将 Llama2-7B 的上下文从 4K 扩到 100K；
- 核心技术：**shifted sparse attention**（Figure 2）+ **位置插值**（Chen+ 2023）；
- 训练数据：**PG-19（书）** 和 **Proof-Pile（数学）**——因为书和长数学证明天然有长距离依赖。

> 数据层面的总结：**做长上下文需要的是"具有长距离依赖"的数据**。书和数学是最常见的来源，也可以合成。

## 5. 课程总结

1. **数据不是从天上掉下来的**——你必须动手去拿。整个流程是：
   > **Live service → Raw snapshot → Processed data（转换、过滤、去重）**
2. **数据是语言模型差异化的关键**。架构已经高度通用，模型之间的差异主要来自数据。
3. **法律与伦理问题不可回避**——版权、隐私、ToS。
4. **这个领域目前还非常"启发式"**——规则混搭经验，大量改进空间。

## 附录：贯穿整节课的几个核心思维工具

1. **质量的代理（Surrogates for Quality）**：
   - Wikipedia 本身 → Wikipedia 引用的页面 → 指令数据（OpenHermes, ELI5）
   - "质量"从没有公认定义，所以用已知的高质量池当"正样本"。

2. **规则过滤 vs 模型过滤的钟摆**：
   - 早期（C4, Gopher, RefinedWeb, Dolma）：拒绝用弱模型做过滤，怕引入偏差；
   - 现在（DCLM, Nemotron-CC, OLMo 2）：模型过滤赢了——benchmark 提升太多，不用就亏。

3. **去重（Deduplication）的三层演进**：
   - **精确去重**（CCNet 的段落归一化）；
   - **MinHash + Jaccard** 近似去重（RefinedWeb、FineWeb）；
   - **Bloom Filter**（Dolma）。

4. **"数据质量"和"数据数量"的张力**：
   - DCLM 激进过滤 → 3.8T tokens；
   - Nemotron-CC 为了更多 token 换工具、加合成改写 → 6.3T；
   - 大模型训练需要的 token 数（15T–36T）对过滤压力提出新要求。

5. **合成数据的地位**：从可选 → 必选。
   - 低质量数据：让 LM 改写；
   - 高质量数据：让 LM 生成任务（QA、摘要、抽取等）；
   - 后训练全靠合成。

6. **Live service 的消亡**：
   - 2023 年前：Reddit、StackExchange 数据可以免费拿；
   - 2023 年后：平台意识到自己在被白嫖 → 商用必须付费。
   - 这是"数据护城河"形成的关键节点。
